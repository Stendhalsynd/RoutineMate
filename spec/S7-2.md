# S7-2 Firestore Schema And Migration

## Goal
- Define the canonical Firestore document model for RoutineMate and add a repeatable Notion-to-Firestore migration pipeline with parity reporting.

## Scope
- Add shared Firestore schema helpers for `users/{uid}` and user-scoped subcollections.
- Add a server/admin Firebase helper for scripts and backend jobs.
- Add migration planning utilities that transform Notion-origin entities into deterministic Firestore writes.
- Add a migration script with `--dry-run` support and optional per-user filtering.
- Add parity-focused tests proving IDs, dates, delete metadata, and per-entity counts are preserved.

## Acceptance Criteria
1. Firestore schema helpers expose the canonical user root path and supported subcollection names in code.
2. Server-side Firebase helpers can initialize `firebase-admin` from env vars without hardcoding secrets.
3. Migration planning preserves existing record IDs, user ownership, created dates, and soft-delete metadata.
4. The migration script can print a dry-run parity report before writing to Firestore.
5. Failing-then-passing tests prove schema paths and migration parity for sessions, meals, workouts, body metrics, goals, templates, and reminder settings.

## Verification
- `node --test --import tsx tests/unit/firestore-migration.test.ts`
- `npm run typecheck`
