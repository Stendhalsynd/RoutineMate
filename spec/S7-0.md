# S7-0 Ralph Execution Charter

## Goal
- Lock the S7 execution model for the Firebase migration, full data cutover, Neo Brutalism redesign, and release automation work.
- Ensure every S7 slice is traceable through spec, failing tests, implementation, review, verification, and release evidence.

## Required Workflow
1. Work one numbered slice at a time.
2. Create or update the matching `plan/` progress log before code changes.
3. Write at least one failing test that proves the slice is not complete yet.
4. Implement the minimum code needed to pass.
5. Refactor without weakening coverage.
6. Run a code-review gate after each slice.
7. If review finds issues, keep the work inside the same slice until the review passes.
8. Do not move to the next slice until verification evidence is recorded.

## Completion Promise
- `ralph` mode stays active until all S7 slices are complete or an external blocker makes progress impossible.
- The final completion bar is:
  - all numbered slices complete
  - zero pending or in-progress tasks in `plan/`
  - fresh tests and builds pass
  - architect-level verification passes
  - Android release APK is rebuilt
  - GitHub release is updated
  - current branch is pushed to `origin`
  - Discord deployment summary is sent

- `S7-9` Mobile Neo Brutalism redesign

## Review Gate
- Every slice must be reviewed with a `code-review` mindset that prioritizes behavioral regressions, logic defects, missing tests, and release risks.
- Review findings create follow-up work inside the same slice rather than deferring to a future slice.

## Verification Rules
- Foundation and data slices must include type, integration, and migration evidence.
- UI slices must include regression evidence for the affected screen behavior.
- Release slices must verify the actual artifact or external target, not just local scripts.
